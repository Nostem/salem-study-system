Where the Council Agrees

  • Semantic search, QMD, and YAML queries are not worth building. Every advisor either explicitly rejected these or ignored them. The vocabulary is narrow and precise — not the fuzzy domain where embeddings help.
  • The NRC exam is closed-book, so the core problem is retention, not retrieval. Four of five advisors converged on this. The 177+ ingested exam questions are a passive asset that should be made active.
  • The structured annotation data already exists. Callout-exam blocks, hi-exam spans, and four-option justification cards need harvesting and surfacing, not re-authoring.
  • K/A frequency analysis is low-cost, high-value. A simple count across exam years reveals which areas Salem examiners hit repeatedly.

Where the Council Clashes

Is search a solved problem or a scaling bottleneck?

The Contrarian says Pagefind is good enough. Peer reviewers push back: at 1000 articles, Pagefind returns 40 results for "RHR" with no way to filter by type. Resolution: Pagefind faceted filtering solves this with minimal effort — add data-pagefind-filter attributes. No new search engine needed.

Build SRS in-site or export to Anki?

First Principles wants in-site spaced repetition. The Contrarian warns it's a maintenance trap on a static site. Resolution: Export to Anki. Don't reinvent SM-2 scheduling and localStorage persistence. The wiki's job is to be the source of flashcards, not the flashcard engine.

Recall of facts vs. discrimination between confusable facts?

NRC exams test application and discrimination ("which of these four setpoints applies here?"), not isolated recall. Resolution: The Anki export should include all four choices and justifications, not just the correct answer. This trains discrimination, not memorization.

Blind Spots the Council Caught

  • Harvest structured annotations site-wide. A "show me every hi-exam span" view gives a cross-article exam-relevant fact index for free.
  • Graph-based coverage analysis. The existing wikilink graph can answer "which systems have no exam questions?" and "which articles are orphans?" — quality/completeness audit tool.
  • Source traceability queries. "Show me every article referencing TS 3/4.5" serves both cross-cutting study and wiki integrity verification.
  • Wrong-answer drill mode. The four-choice justification content supports "why is each wrong answer wrong?" training — higher-order than flashcard recall.

The Recommendation

Build three things, in this order:

  1. Anki export pipeline — A build-time script that harvests callout-exam blocks and exam question cards, converts to Anki-importable format with tags (K/A area, exam year, system, source article). Include all four options and justifications so cards train discrimination. Do NOT build SRS into the Astro site.
  2. Pagefind faceted filtering — Add data-pagefind-filter attributes for article type, K/A group, exam year, and unit. Small template change, massive usability gain at scale.
  3. K/A frequency heatmap — A script or Astro page that parses K/A tags across all exam years and produces a visual heatmap. Direct study prioritization from data.

Do NOT build: semantic search, QMD, YAML queries, custom SRS, or scenario-driven search.

The One Thing to Do First

Build the Anki export script. Write a Python script in scripts/ that walks every .md file in wiki/, extracts callout-exam blocks and exam question cards (the existing four-choice format with justifications), tags each with K/A area and exam year, and writes an Anki-importable file. Import the deck and start drilling today. Every day without active recall on the 177+ questions already ingested is a day studying in passive-reading mode when the exam tests active discrimination.

Full Advisor Responses

The Contrarian

The fatal flaw: you're asking about search when the real bottleneck is retrieval under exam pressure, not at a keyboard. NRC exam is closed-book. Pagefind is already good enough. Semantic search adds massive complexity for marginal gain. Spaced repetition on a static site becomes a maintenance trap. The callout-exam blocks are the right primitive but they're trapped in articles. Build a standalone drill mode — sequential, timed, browser-only. When did you last fail to find something versus fail to recall it under pressure?

The First Principles Thinker

The question being asked is "what search features?" The question that matters is "what makes candidates fail?" NRC exams test recall, not retrieval. You have 177 exam questions — that's a flashcard deck. Build one thing: spaced repetition that surfaces callout-exam blocks as active recall prompts, tracks wrong answers, and requeues them. Pagefind is sufficient for reference. Cross-exam pattern analysis can be done in an afternoon by counting K/A tags.

The Expansionist

The flashcard angle is undervalued — 177 questions with justifications is a complete Anki deck. Cross-exam K/A frequency analysis is the real goldmine: surface which K/A areas Salem tests repeatedly across 6+ exam years. Question clustering by correct-answer pattern reveals personal weak spots. Better Pagefind metadata tags beat semantic search. The 1000-article milestone is where pattern analysis becomes statistically meaningful.

The Outsider

The system is invisible to anyone who doesn't already know nuclear ops. NRC exams give scenarios requiring three domains simultaneously — tech spec, system behavior, procedure. Pagefind finds text, not relationships. Most practical: scenario-driven cross-linking in search results. Wrong-answer analysis is highest leverage — surface "what did the exam expect operators to confuse with X?" Spaced repetition is useful but only after you solve retrieval.

The Executor

Three concrete steps: 1) Anki export script converting callout-exam blocks to Anki-importable format. 2) K/A frequency heatmap — 30-line Python script parsing K/A occurrences across exam years. 3) Pagefind faceted filtering — add data-pagefind-filter attributes for article type, K/A group, exam year. Skip semantic search, QMD, YAML queries. Fix retention first, study prioritization second, search discoverability third.
Peer Review Highlights
Key consensus: Executor's concrete deliverables were rated strongest by 2 of 5 reviewers. Outsider's multi-domain integration insight rated strongest by 2 of 5. All reviewers noted the existing structured annotations (callout-exam, hi-exam) are an untapped data source.

Key disagreement: Whether search or recall is the primary bottleneck. Resolved by faceted Pagefind (search improves without new infrastructure) plus Anki export (recall solved by mature external tool).

Novel insight from reviews: Graph traversal queries for coverage auditing — surface which systems have no exam questions linked, which articles are orphans. Source traceability queries for wiki integrity verification.
